# 最终产品测试指导

版本日期：2026-06-17

## 文档目的

本文档用于指导最终产品生产测试、功能验证和回归测试。

后续每确定一个模块的工作逻辑，都必须同步维护本文档中的测试方法、测试步骤和判定标准。

## 通用测试要求

### 当前固件阶段说明

截至当前版本，固件已完成主循环骨架、板级 IO 适配、PC15 运行心跳、DC_CTRL 输入确认、O 类异常置位、ALARM 输出、蜂鸣器等级输出、通道请求解析、继电器 500ms 非阻塞脉冲、500ms 反馈等待、B~J 反馈异常置位、三路 NTC 温度换算、风扇 PWM 自动切换、K~M 温度异常置位和 OLED 基础显示。

当前版本可以先烧录进行以下低风险硬件测试：

- `PC15/RUN_HB` 10Hz 运行心跳。
- `DC_CTRL` 外部控制电源异常检测。
- O 类异常触发和解除。
- `ALARM` 低有效持续输出。
- `BEEP` 严重故障长鸣输出。
- 风扇 PWM 初始化为 50%。
- 单通道 K_EN 稳定输入后的继电器 ON/OFF 500ms 脉冲。
- 继电器动作后 500ms 反馈等待。
- B~J 反馈异常触发和自动解除。
- 多路 K_EN 同时有效时的 A 类异常。
- 打开状态下直接请求另一路时的 N 类异常。
- 三路 NTC 温度显示数据是否合理。
- 最高温度控制风扇 50%/95% PWM 切换。
- K~M 温度异常触发和解除。
- NTC 断线或短路时对应 K~M 异常和风扇强制 95%。
- NTC 断线或短路出现后需要锁存异常状态；ADC 接近有效 RT 表边界或出现明显漂浮跳变时均视为 NTC 异常，连续 3 次采样恢复正常后才允许退出风扇 95%。
- OLED 上电 LOGO、自检进度、主界面和报警摘要显示。
- OLED 三通道状态反显、三路温度和风扇 PWM 显示。
- OLED 风扇行显示 PWM 和 FAN_SEN 换算 RPM。

当前版本暂不用于以下测试：

- 日志分页和 Flash 记录。
- LOGO 位图进一步美化。
- 看门狗生产策略。

### 测试前准备

- 确认控制板供电正常。
- 确认外部 DC24V 控制电源接入正常。
- 确认 OLED 显示屏连接正常。
- 确认 ALARM 输出和蜂鸣器接线正常。
- 确认三路外部使能输入可独立控制。
- 确认三路继电器反馈和接触器反馈可测量。
- 确认调试串口可用。

### 通用判定原则

- 任何异常测试都必须确认 ALARM 输出是否符合低电平有效规则。
- ALARM 不区分异常等级，只要存在任意 A~O 异常，必须持续输出低电平。
- 所有异常解除后，ALARM 必须恢复高电平。
- 蜂鸣器需要按当前最高异常等级验证报警时序。
- B~J 紧急故障蜂鸣器必须为 100ms 响、100ms 停循环。
- K~M 一般故障蜂鸣器必须为 500ms 响、500ms 停循环。
- 最高异常等级变化时，蜂鸣器必须立即切换时序，并从响声开始。
- 所有异常解除后，蜂鸣器必须立即停止。
- 任何继电器动作测试都必须确认输出脉冲宽度为 500ms。
- 任何继电器动作后必须等待 500ms 再判定反馈。
- K_EN 触发测试必须确认 150ms 间隔连续 3 次稳定后才生效，确认时间约 450ms。
- DC_CTRL 测试必须确认 50ms 间隔连续 3 次稳定后才确认异常或恢复，确认时间约 150ms。
- DC_CTRL 的采样和判定必须早于 K_EN。
- DC_CTRL 异常疑似出现时，必须立即冻结 K_EN 控制事件。
- 每次测试后必须确认系统可以恢复到明确状态。

## 1. 上电启动测试

### 测试目的

验证系统上电后 LOGO、自检、待机或接管流程是否正确。

### 测试步骤

1. 确认 `K1_EN`、`K2_EN`、`K3_EN` 全部为高电平。
2. 确认所有继电器反馈和接触器反馈均为低电平。
3. 给系统上电。
4. 观察 OLED。
5. 观察 ALARM 和蜂鸣器。

### 期望结果

- OLED 显示 LOGO，持续 2 秒。
- LOGO 显示期间主循环持续运行，不输出继电器动作，不触发业务报警结论。
- 随后进入 3 秒自检进度显示。
- 自检期间显示进度条和当前自检项目。
- 自检通过后进入待机界面。
- ALARM 保持正常状态。
- 蜂鸣器无异常报警。

## 1.1 应用状态机调度测试

### 测试目的

验证 `app_controller` 在 LOGO、自检、待机、运行和报警状态下均采用非阻塞调度。

### 测试步骤

1. 上电后观察 LOGO 2 秒和自检 3 秒期间的主循环诊断信息。
2. 在 LOGO 显示期间改变 K_EN 输入，观察是否不会输出继电器动作。
3. 在自检期间保持外部请求与反馈一致，观察是否接管运行状态。
4. 制造一个 A 类或 N 类异常，观察是否进入报警状态。
5. 解除异常后，观察系统是否基于当前稳定输入和反馈重新接管。

### 期望结果

- LOGO、自检、继电器等待和日志分页期间均不阻塞主循环。
- LOGO 阶段不发布继电器动作请求。
- 自检阶段可以识别待机、运行接管、安全关断或 N 类异常。
- 报警状态不等同于软件错误状态，A~O 业务异常不停止看门狗。
- 异常解除后不补执行旧请求。

## 2. 上电状态识别与安全复位测试

### 测试目的

验证磁保持继电器掉电保持后，系统能否根据外部使能输入正确处理残留状态。

### 场景 A：全关断正常待机

测试条件：

- `K1_EN=1`
- `K2_EN=1`
- `K3_EN=1`
- 所有反馈均为低电平。

期望结果：

- 系统进入待机。
- 不输出继电器动作脉冲。
- 不报警。

### 场景 B：外部无使能但一路继电器残留打开

测试条件：

- `K1_EN=1`
- `K2_EN=1`
- `K3_EN=1`
- `K1_1_STA=1`
- `K1_2_STA=1`
- `SW1_STA=1`
- 其他反馈为低电平。

测试步骤：

1. 给系统上电或复位。
2. 观察是否执行 `K1_1_OFF` 和 `K1_2_OFF` 低电平脉冲。
3. 测量脉冲宽度。
4. 等待反馈复查。

期望结果：

- 系统执行通道 1 安全关断。
- `K1_1_OFF` 和 `K1_2_OFF` 输出 500ms 低电平脉冲。
- 500ms 后复查反馈。
- 若反馈全部变低，进入待机。
- 若反馈未变低，触发 N 类异常。

### 场景 C：外部无使能但多路反馈打开

测试条件：

- `K1_EN=1`
- `K2_EN=1`
- `K3_EN=1`
- 两路或三路通道反馈为打开状态。

期望结果：

- 系统触发 N 类异常。
- 禁止自动继电器动作。
- ALARM 输出低电平。
- 蜂鸣器长鸣。

## 3. 看门狗复位接管测试

### 测试目的

验证运行中 MCU 复位后，系统不会误关断外部仍然请求工作的通道。

### 测试条件

- `K1_EN=0`
- `K2_EN=1`
- `K3_EN=1`
- 通道 1 反馈为打开。

### 测试步骤

1. 模拟系统运行在通道 1。
2. 触发 MCU 复位。
3. 保持外部 `K1_EN=0` 不变。
4. 观察重启后的继电器输出、报警和显示。

### 期望结果

- 系统识别外部仍请求通道 1。
- 系统识别通道 1 反馈与请求一致。
- 不输出 ON 或 OFF 脉冲。
- 不报警。
- 接管当前运行状态。

## 4. 通道开启测试

### 测试目的

验证单通道开启流程、互锁检查和反馈检查。

### 以通道 1 为例

测试条件：

- `K1_EN` 初始为高电平。
- `K2_EN=1`
- `K3_EN=1`
- 所有反馈为低电平。

测试步骤：

1. 将 `K1_EN` 拉低。
2. 保持低电平稳定。
3. 确认系统完成 150ms 间隔连续 3 次确认。
4. 观察 `K1_1_ON` 和 `K1_2_ON` 输出。
5. 测量低电平脉冲宽度。
6. 等待反馈检查。

期望结果：

- `K1_1_ON` 和 `K1_2_ON` 同时输出 500ms 低电平脉冲。
- `K1_1_ON` 和 `K1_2_ON` 的起始时间应一致。
- `K1_1_ON` 和 `K1_2_ON` 的结束时间应一致。
- 其他通道和其他方向的继电器输出保持高电平。
- 脉冲后 500ms 检查 `K1_1_STA`、`K1_2_STA`、`SW1_STA`。
- `K1_1_STA`、`K1_2_STA`、`SW1_STA` 均为高电平，且其他六个反馈点均为低电平时，通道 1 显示为打开。
- 不产生报警。

## 5. 通道关闭测试

### 测试目的

验证单通道关闭流程和反馈检查。

### 以通道 1 为例

测试条件：

- `K1_EN=0`
- 通道 1 反馈为打开。

测试步骤：

1. 将 `K1_EN` 拉高。
2. 保持高电平稳定。
3. 确认系统完成 150ms 间隔连续 3 次确认。
4. 观察 `K1_1_OFF` 和 `K1_2_OFF` 输出。
5. 测量低电平脉冲宽度。
6. 等待反馈检查。

期望结果：

- `K1_1_OFF` 和 `K1_2_OFF` 同时输出 500ms 低电平脉冲。
- `K1_1_OFF` 和 `K1_2_OFF` 的起始时间应一致。
- `K1_1_OFF` 和 `K1_2_OFF` 的结束时间应一致。
- 其他通道和其他方向的继电器输出保持高电平。
- 脉冲后 500ms 检查 `K1_1_STA`、`K1_2_STA`、`SW1_STA`。
- 九个反馈点均为低电平时，通道 1 显示为关闭。
- 不产生报警。

## 6. 继电器动作状态机测试

### 测试目的

验证继电器动作层采用非阻塞状态机，动作忙碌期间不会接受新的继电器动作命令。

### 测试步骤

1. 触发通道 1 开启动作。
2. 在 `K1_1_ON` 和 `K1_2_ON` 输出 500ms 低电平期间，尝试触发通道 2 开启动作。
3. 在通道 1 脉冲结束后的 500ms 反馈等待期间，再次尝试触发通道 2 开启动作。
4. 观察所有继电器输出。

### 期望结果

- 通道 1 动作期间，通道 2 不输出 ON 脉冲。
- 通道 1 反馈等待期间，通道 2 不输出 ON 脉冲。
- 忙碌期间的新动作命令被拒绝，但不直接触发报警。
- 状态机完成通道 1 动作流程后，才允许上层重新解析并发起新动作。
- 任意时刻无关继电器输出保持高电平。

## 7. 多路使能冲突测试

### 测试目的

验证 A 类严重故障。

### 测试步骤

1. 同时拉低任意两路 `K_EN`。
2. 保持稳定。
3. 观察报警输出、蜂鸣器和继电器动作。
4. 将三路 `K_EN` 全部恢复为高电平。
5. 确认 A 类异常解除。
6. 再重新拉低一路 `K_EN`。

### 期望结果

- 触发 A 类异常。
- 禁止继电器动作。
- ALARM 输出低电平。
- 蜂鸣器长鸣。
- OLED 显示 A 类异常。
- A 类异常解除后，不自动执行异常期间曾经出现过的旧请求。
- 重新稳定给出单路 `K_EN` 后，系统才允许生成新的通道请求。

## 8. 通道切换互锁测试

### 测试目的

验证系统不允许通道间直接自动切换，必须先关闭当前通道，再开启另一路通道。

### 场景 A：合法切换流程

测试条件：

- 通道 1 已打开。
- `K1_EN=0`
- `K2_EN=1`
- `K3_EN=1`

测试步骤：

1. 将 `K1_EN` 拉高。
2. 等待系统完成 150ms 间隔连续 3 次确认。
3. 确认通道 1 执行关闭动作。
4. 确认通道 1 反馈全部变为关闭。
5. 将 `K2_EN` 拉低。
6. 等待系统完成 150ms 间隔连续 3 次确认。
7. 观察通道 2 开启动作和反馈。

期望结果：

- 通道 1 先关闭。
- 通道 1 反馈关闭确认后，通道 2 才允许打开。
- 通道 1 和通道 2 不出现同时打开。
- 不产生报警。

### 场景 B：非法直接切换

测试条件：

- 通道 1 已打开。
- `K1_EN=0`
- `K2_EN=1`
- `K3_EN=1`

测试步骤：

1. 不等待通道 1 关闭，直接将 `K1_EN` 拉高并将 `K2_EN` 拉低。
2. 保持信号稳定。
3. 观察继电器动作、报警输出和显示。

期望结果：

- 系统不自动从通道 1 切换到通道 2。
- 禁止通道 2 开启动作。
- 当前通道仍处于打开反馈时，直接请求另一路通道应触发 N 类严重故障。
- ALARM 输出低电平。
- 蜂鸣器长鸣。

恢复步骤：

1. 将 `K1_EN`、`K2_EN`、`K3_EN` 全部恢复为高电平。
2. 等待系统关闭当前通道。
3. 确认所有反馈均为关闭状态。
4. 确认 N 类异常解除。
5. 再单独拉低目标通道 `K_EN`。

恢复期望结果：

- N 类异常解除前，系统不接受新的开通道请求。
- N 类异常解除后，重新稳定给出的单路请求可以正常执行。

### 场景 C：动作忙碌期间新请求

测试条件：

- 通道 1 已打开。
- `K1_EN=0`
- `K2_EN=1`
- `K3_EN=1`

测试步骤：

1. 将 `K1_EN` 拉高，使系统开始关闭通道 1。
2. 在通道 1 OFF 脉冲或反馈等待期间，将 `K2_EN` 拉低。
3. 观察通道 2 是否立即打开。
4. 等待通道 1 关闭反馈确认完成。
5. 保持 `K2_EN` 为低电平，观察系统后续动作。

期望结果：

- 通道 1 关闭动作未完成前，通道 2 不允许打开。
- 系统不缓存忙碌期间的旧动作请求。
- 通道 1 反馈确认关闭后，系统基于最新稳定输入重新解析请求。
- 如果此时只有 `K2_EN=0`，系统允许按正常流程打开通道 2。

## 9. 反馈异常测试

### 测试目的

验证 B~J 类紧急故障，以及反馈异常后的指令锁定保护。

### 测试步骤

1. 执行某一路通道开启或关闭。
2. 人为制造某一路继电器反馈或接触器反馈不符合预期。
3. 等待反馈检查。
4. 在反馈异常未解除时，尝试给出新的通道动作请求。
5. 恢复所有反馈到正常一致状态。

### 期望结果

- 对应 B~J 异常被置位。
- ALARM 输出低电平。
- 蜂鸣器 100ms 响、100ms 停循环。
- OLED 显示对应异常。
- 反馈异常未解除前，系统不接受新的通道动作指令。
- 三个通道反馈恢复正常一致状态后，B~J 异常允许自动解除。
- B~J 异常解除后，不自动执行异常期间曾经出现过的旧请求。
- 重新稳定给出单路 `K_EN` 后，系统才允许生成新的通道请求。

### 异常解除后接管测试

场景 A：恢复后一致接管

1. 触发通道 1 反馈异常。
2. 保持 `K1_EN=0`。
3. 恢复九路反馈到通道 1 正常打开状态。

期望结果：

- B~J 异常自动解除。
- 系统接管通道 1 当前打开状态。
- 不重复输出 `K1_1_ON` 和 `K1_2_ON` 脉冲。

场景 B：恢复后变成另一路请求

1. 触发通道 1 反馈异常。
2. 异常期间将外部输入改为 `K2_EN=0`。
3. 恢复九路反馈到通道 1 正常打开状态。

期望结果：

- 系统不自动切换到通道 2。
- 系统按非法直接切换处理。
- 触发 N 类严重故障。

### 打开反馈检查

以通道 1 为例：

- 打开通道 1 后，`K1_1_STA=1`、`K1_2_STA=1`、`SW1_STA=1`，且其他六个反馈点全部为低电平才算成功。
- `K1_1_STA` 未变为高电平时，触发 B 类异常。
- `K1_2_STA` 未变为高电平时，触发 E 类异常。
- `SW1_STA` 未变为高电平时，触发 H 类异常。
- 其他通道任意反馈点误为高电平时，触发该反馈点对应的 B~J 异常。

通道 2、通道 3 按对应 B~J 映射测试。

### 关闭反馈检查

以通道 1 为例：

- 关闭通道 1 后，九个反馈点全部为低电平才算成功。
- `K1_1_STA` 未变为低电平时，触发 B 类异常。
- `K1_2_STA` 未变为低电平时，触发 E 类异常。
- `SW1_STA` 未变为低电平时，触发 H 类异常。

通道 2、通道 3 按对应 B~J 映射测试。

## 10. 温度异常测试

### 测试目的

验证 K~M 类一般故障和风扇控制。

### 测试步骤

1. 模拟三路 NTC 温度均低于 33℃。
2. 模拟三路温度中的最高值大于等于 35℃。
3. 模拟三路温度中的最高值下降到 33℃ 以下。
4. 模拟 NTC 温度大于等于 60℃。
5. 模拟温度下降到 58℃ 以下。
6. 模拟 NTC 断线、短路或换算失败。
7. 在 K~M 异常存在期间，执行一次通道开启或关闭请求。
8. 模拟风扇测速异常。

### 期望结果

- 三路最高温度低于 33℃ 时，风扇 PWM 为 50%。
- 三路最高温度大于等于 35℃ 时，风扇 PWM 为 95%。
- 温度从 35℃ 附近下降但未低于 33℃ 时，风扇 PWM 保持 95%，不频繁切换。
- 温度大于等于 60℃ 后，按 500ms 周期连续 3 次确认，触发对应 K~M 异常。
- K~M 异常时 ALARM 输出低电平。
- K~M 异常时蜂鸣器 500ms 响、500ms 停循环。
- 温度下降到 58℃ 以下后，按 500ms 周期连续 3 次确认，对应异常解除。
- NTC 断线、短路或换算失败时，归入对应 K/L/M 异常，并显示温度传感器异常提示。
- NTC 断线、短路、ADC 漂浮或换算失败一旦被采样到，风扇必须保持 95%，不得在 95% 和 50% 间来回跳变。
- K~M 异常期间，通道开启和关闭仍按外部请求正常执行。
- K~M 异常不会主动关闭已经打开的通道。
- 风扇测速异常不触发 A~O 异常，不影响 ALARM 和 BEEP，仅作为诊断信息或显示提示。

## 11. 多异常优先级测试

### 测试目的

验证多个异常同时存在时，异常位不互相覆盖，ALARM 和蜂鸣器输出符合规则。

### 测试步骤

1. 先触发 K 类温度异常。
2. 保持 K 类异常存在，再触发 B 类反馈异常。
3. 保持 B 类和 K 类异常存在，再触发 A 类多路使能冲突。
4. 分别解除 A 类、B 类、K 类异常。

### 期望结果

- A、B、K 异常可以同时存在，不互相覆盖。
- 任意异常存在期间，ALARM 持续输出低电平。
- 多个异常同时存在时，蜂鸣器按最高异常等级输出。
- A 类存在时蜂鸣器长鸣。
- A 类解除但 B 类仍存在时，蜂鸣器切换为 100ms 响、100ms 停循环。
- B 类解除但 K 类仍存在时，蜂鸣器切换为 500ms 响、500ms 停循环。
- 所有异常均解除后，ALARM 恢复高电平，蜂鸣器停止。
- 蜂鸣器每次等级切换后，都应立即从响声状态重新开始计时。
- 所有异常解除后，BEEP 应立即恢复高电平。

## 12. 外部 DC24V 电源异常测试

### 测试目的

验证 O 类严重故障，以及 PLC 控制源异常时 K_EN 控制权冻结。

### 测试步骤

1. 正常接入 DC24V，确认 `DC_CTRL=0`。
2. 断开 DC24V，确认 `DC_CTRL=1`。
3. 观察报警输出、蜂鸣器和 OLED。
4. 恢复 DC24V。

### 期望结果

- DC24V 正常时无 O 类异常。
- DC24V 丢失时触发 O 类异常。
- ALARM 输出低电平。
- 蜂鸣器长鸣。
- OLED 显示 O 类异常。
- DC24V 恢复后，O 类异常解除。

## 13. PLC 掉电保持测试

### 测试目的

验证 PLC 掉电导致 K_EN 丢失时，系统不会跟随 K_EN 丢失而关断磁保持继电器。

### 测试条件

- 通道 1 已正常打开。
- `K1_EN=0`
- `K2_EN=1`
- `K3_EN=1`
- 通道 1 反馈为打开。

### 测试步骤

1. 断开外部 PLC 主供电，使 `DC_CTRL` 变为高电平。
2. 同时观察 `K1_EN` 是否因 PLC 掉电变为高电平。
3. 观察继电器 OFF 输出。
4. 观察 ALARM、蜂鸣器和 OLED。

### 期望结果

- DC_CTRL 疑似异常出现时，系统立即冻结 K_EN 控制事件。
- DC_CTRL 按 50ms 间隔连续 3 次确认异常后，触发 O 类严重故障。
- 即使 `K1_EN` 变为高电平，也不输出 `K1_1_OFF` 和 `K1_2_OFF`。
- 通道 1 磁保持继电器保持原状态。
- ALARM 输出低电平。
- 蜂鸣器长鸣。
- OLED 显示 PLC 电源异常或外部控制源异常。

## 14. PLC 电源恢复重同步测试

### 测试目的

验证 PLC 电源恢复后，系统不会立即盲目执行 K_EN，而是重新同步外部请求和内部反馈。

### 场景 A：恢复后一致接管

测试条件：

- PLC 掉电期间通道 1 保持打开。
- PLC 恢复后 `K1_EN=0`。
- 通道 1 反馈仍为打开。

期望结果：

- DC_CTRL 稳定恢复低电平后，系统解除 O 类异常。
- 系统接管通道 1 当前运行状态。
- 不输出 ON 或 OFF 脉冲。

### 场景 B：恢复后外部无请求但通道残留打开

测试条件：

- PLC 恢复后 `K1_EN=1`、`K2_EN=1`、`K3_EN=1`。
- 某一路通道反馈仍为打开。

期望结果：

- 系统执行该通道安全关断。
- OFF 输出为 500ms 低电平脉冲。
- 反馈复查成功后进入待机。

### 场景 C：恢复后请求和反馈冲突

测试条件：

- PLC 恢复后 `K2_EN=0`。
- 当前反馈显示通道 1 打开。

期望结果：

- 系统不自动切换。
- 触发异常。
- 禁止继电器动作。

## 15. OLED 显示测试

### 测试目的

验证显示内容和界面分区。

### 测试项目

- LOGO 显示 2 秒。
- 第一屏只显示旧版大 LOGO，不显示 `3CH CTRL V5` 或其他说明文字。
- OLED 冷上电时允许短暂等待屏幕电源稳定，OLED 应在自动初始化成功后显示 LOGO，不应需要按复位键才显示。
- OLED 初始化失败时应自动重试，不阻塞主循环。
- LOGO 显示期间不输出继电器动作。
- 自检进度显示 3 秒。
- 自检界面显示 `minyerlogo.c` 中的小 LOGO 或品牌标识。
- 自检界面小 LOGO 不遮挡进度条和自检项目文本。
- 自检界面显示进度条。
- 自检界面不显示 `SELF TEST` 和 `STEP`，只显示当前实际自检项目，使用简短英文或缩写。
- 自检界面进度条与自检项目文字之间必须留出间距，不能贴边或重合。
- 待机界面显示三分区：
  - 异常信息区。
  - 三通道状态区。
  - 温度和风扇转速区。
- OLED 页面不使用中文显示。
- OLED 运行界面同时显示三路温度。
- OLED 运行界面三路温度必须显示在同一行。
- OLED 运行界面风扇信息同时显示 PWM 和 RPM。
- 报警区域、通道状态区域、温度和风扇区域之间使用横线分隔，横线不得压住字符。
- 报警区域预留两行字符空间。
- 三通道状态区分为三个位置，分别居中显示 `CH1`、`CH2`、`CH3` 和状态。
- 实际反馈确认打开的通道使用整个通道方框反显。
- 实际反馈确认关闭的通道只显示正常字符，不显示方框。
- 外部 K_EN 请求未完成反馈确认前，不应单独改变通道格反显状态。
- 继电器动作中和反馈等待中的短暂状态不要求单独显示。
- 异常发生时显示对应异常信息。
- `NTC ERR` 提示显示在报警区域右端，不占用温度和风扇信息区域。
- 异常界面保留通道状态、三路温度和风扇状态摘要。
- 多个异常同时存在时，优先显示最高等级异常。
- 同等级多个异常同时存在时，每 1 秒轮询显示一个异常。
- OLED 应采用脏区刷新机制，普通状态更新不应出现明显全屏闪动。
- 通道状态变化时，应只刷新通道状态区或相关局部区域。
- 温度或风扇信息变化时，应只刷新温度和风扇信息区或相关局部区域。
- 异常信息变化或轮询时，应只刷新异常信息区或相关局部区域。

## 16. 看门狗策略测试

### 测试目的

验证看门狗只处理软件卡死或主循环失控，不因 A~O 业务异常直接复位。

### 测试项目

#### 调试阶段

- 确认调试配置下看门狗默认关闭。
- 确认断点调试不会因为看门狗导致复位。

#### 生产阶段

- 将 `APP_WATCHDOG_ENABLE` 配置为 1，启用生产看门狗。
- 确认看门狗由 `watchdog_manager` 集中喂狗。
- 确认其他模块没有直接喂狗逻辑。
- 确认正常主循环运行时系统不会复位。
- 确认 LOGO 显示和自检进度期间仍由主循环正常调度并喂狗。

#### 业务异常不复位

测试步骤：

1. 分别触发 A、B、K、N、O 类异常。
2. 保持异常存在，观察系统是否因看门狗复位。

期望结果：

- A~O 业务异常本身不直接停止喂狗。
- 系统不因业务异常反复复位。
- 对应异常仍通过 ALARM、蜂鸣器、动作禁止或输入冻结处理。

#### 主循环异常复位

测试步骤：

1. 人为模拟主循环卡死或调度器长时间不返回。
2. 人为模拟关键模块长时间未被调度。
3. 观察看门狗是否触发复位。

期望结果：

- 主循环卡死后，看门狗应触发系统复位。
- 关键模块长时间未调度时，应停止喂狗并触发复位。
- 复位后系统记录或显示上次复位原因为看门狗复位，例如显示 `WDT RST`。
- 看门狗复位原因只作为诊断信息，不应触发 A~O 异常，不应拉低 ALARM，不应触发蜂鸣器。
- 重启后系统仍按上电/复位接管策略识别外部请求和反馈状态。

#### 时间约束

- 单次主循环耗时超过 50ms 时，应记录诊断信息。
- 单次主循环耗时超过 200ms 时，应作为调度异常依据。
- LOGO 2 秒、自检 3 秒、继电器 500ms 脉冲和 500ms 反馈等待均不得使用阻塞延时实现。
- 在上述长时序过程中，主循环必须持续运行，不能影响 500ms 集中喂狗周期。

## 17. 维护按键和日志测试

### 测试目的

验证 KEY1、KEY2 只用于日志调出、分页、退出和清空，不影响蜂鸣器、ALARM、继电器和当前故障状态。

### 按键测试

测试步骤：

1. 系统正常运行时短按 KEY1。
2. 观察调试串口是否输出日志第一页。
3. 短按 KEY2。
4. 观察调试串口是否输出下一页日志。
5. 再次短按 KEY1。
6. 观察调试串口是否回到上一页日志。
7. 长按 KEY1。
8. 观察是否退出日志分页输出状态。
9. 长按 KEY2。
10. 观察是否执行日志清空。
11. 清空后再次短按 KEY1 调出日志。
12. 观察日志中是否保留 `LOG_CLEARED` 记录。
13. 按下 KEY1 或 KEY2 不足 3 秒后释放。
14. 观察是否未触发长按动作，并且不补触发短按动作。

期望结果：

- KEY1、KEY2 均采用轮询确认，按键抖动不应导致重复触发。
- KEY1、KEY2 长按必须消抖确认后持续满 3 秒才触发。
- 未满 3 秒释放时，不触发长按动作，也不转为短按动作。
- 日志每页最多输出 50 条。
- 日志总记录上限按 2000 条测试，超过上限后新日志覆盖最旧日志。
- 清空日志后应立即生成一条新的 `LOG_CLEARED` 记录。
- 每输出一页后暂停，等待按键翻页。
- 日志输出期间系统主循环继续运行。
- 按键操作不应静音蜂鸣器。
- 按键操作不应清除当前真实存在的 A~O 异常。
- 按键操作不应改变继电器状态。
- PC15 不参与日志按键测试，当前作为 10Hz 运行心跳输出。

## 18. 运行心跳输出测试

### 测试目的

验证 `PC15/RUN_HB` 能向外部 PLC 提供程序主循环运行状态。

### 测试条件

- 使用示波器或频率计测量 `PC15/RUN_HB`。
- 系统已正常上电并进入主循环。

### 测试步骤

1. 给系统正常供电。
2. 使用示波器测量 `PC15/RUN_HB`。
3. 观察输出频率和波形。
4. 复位 MCU，观察复位期间心跳是否中断。

### 期望结果

- 主循环正常运行时，`PC15/RUN_HB` 输出 10Hz 方波。
- 正常波形周期约 100ms，高低电平各约 50ms。
- MCU 复位、程序卡死或主循环停止时，心跳输出应停止或不再保持 10Hz 方波。

### 日志内容测试

测试步骤：

1. 上电启动系统。
2. 执行一次通道打开和关闭。
3. 分别触发并解除至少一个 A 类、一个 B~J 类、一个 K~M 类和一个 O 类异常。
4. 通过 KEY1 调出日志。

期望结果：

- 日志记录上电启动事件。
- 日志只记录实际发生动作的目标通道，例如通道 1 打开或通道 1 关闭。
- 未动作通道的维持状态不写入日志。
- 日志记录异常产生和异常解除事件。
- 日志时间戳使用秒为单位，从本次上电启动开始计时。
- 日志不应每秒固定写入，只应在事件发生时写入。
- 温度正常变化、风扇 PWM 变化和风扇转速变化不应写入日志。
- 温度或风扇相关问题只有触发 A~O 异常时，才按异常事件写入日志。

### 看门狗复位日志测试

测试步骤：

1. 启用生产看门狗。
2. 人为触发一次内部停止喂狗条件。
3. 等待系统由看门狗复位。
4. 重启后通过 KEY1 调出日志。

期望结果：

- 日志记录上一次复位为看门狗复位。
- 如果软件在复位前写入了停止喂狗原因，日志应显示对应软件原因编号。
- 如果无法确认具体软件原因，日志应记录为未知原因。
- 看门狗复位日志不应触发 A~O 异常，不应单独拉低 ALARM，不应单独触发蜂鸣器。
