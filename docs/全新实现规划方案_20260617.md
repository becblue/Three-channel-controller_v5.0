旧代码只作为素材库使用：

- 可以借用 LOGO 字模、OLED 字库、NTC 查表数据、Flash 驱动片段等相对独立内容。
- 不直接继承旧的主循环结构。
- 不直接继承旧的中断/轮询混合逻辑。
- 不直接继承旧的看门狗策略。
- 不直接继承旧的报警判定路径。

新的实现应从原始需求重新设计，先形成清晰的软件架构，再开始写代码。

## 原始需求中的确定内容

### 项目目标

系统是三通道高压切换箱控制系统，核心闭环为：

1. 外部使能输入。
2. 三路互锁判断。
3. 双路磁保持继电器驱动。
4. 双路继电器状态反馈。
5. 接触器状态反馈。
6. 安全报警。
7. OLED 显示。
8. 温度监控和风扇控制。
9. 外部 DC24V 电源异常监控。

### 硬件电平规则

#### K_EN 输入

- `K1_EN -> PB9`
- `K2_EN -> PB8`
- `K3_EN -> PA15`
- 下降沿表示通道开启请求。
- 上升沿表示通道关闭请求。
- 低电平表示该通道请求有效。
- 高电平表示该通道请求无效或关闭。
- `K_EN` 信号不直接驱动业务，必须在 `DC_CTRL` 确认正常后才允许采样确认。
- `K_EN` 采用 150ms 间隔连续 3 次确认，确认时间约 450ms。

#### 继电器输出

- 所有继电器驱动输出低电平有效。
- 初始状态必须为高电平。
- 开启继电器时输出 500ms 低电平脉冲。
- 关闭继电器时输出 500ms 低电平脉冲。
- 脉冲结束后延时 500ms 再检查反馈。

#### 反馈输入

- 继电器状态反馈高电平有效。
- 接触器状态反馈高电平有效。
- 初始正常关断状态应为低电平。

#### 外部 DC24V 电源监控

- `DC_CTRL -> PB5`
- 原始硬件配置为上升沿使能、内部上拉、中断优先级 2。
- `DC_CTRL = 0` 表示 PLC 主供电 DC24V 正常。
- `DC_CTRL = 1` 表示 PLC 主供电 DC24V 丢失或异常。
- 在本项目中，`DC_CTRL` 用于判断外部 PLC 控制源是否有效。
- 当 `DC_CTRL` 异常时，系统必须冻结 `K1_EN`、`K2_EN`、`K3_EN` 控制权，防止 PLC 掉电导致使能信号丢失后误关断磁保持继电器。

#### 报警输出

- `ALARM -> PB4`
- 低电平有效。
- 任意异常激活时持续输出低电平。

#### 蜂鸣器

- `BEEP -> PB3`
- 低电平有效。
- A、N、O 类属于严重故障，蜂鸣器长鸣。
- B~J 类属于紧急故障，蜂鸣器 100ms 响、100ms 停循环。
- K~M 类属于一般故障，蜂鸣器 500ms 响、500ms 停循环。

#### 温度和风扇

- 三路最高温度 >= 35℃：风扇 PWM 95%。
- 三路最高温度 <= 33℃：风扇 PWM 50%。
- 风扇 PWM 切换使用 2℃ 回差。
- 温度 >= 60℃：触发 K、L、M 对应异常。
- 温度下降恢复时使用 2℃ 回差。
- K~M 异常解除阈值为 58℃ 以下。

## 原始文档中的待确认矛盾

### 1. LOGO 显示时间

文档前文和开发需求写 2 秒，OLED 小节写 3 秒。

最终决策：采用 2 秒。

理由：用户已确认采用 2 秒。

### 2. 异常数量

文档有“14种异常类型”的描述，但异常定义明确列出 A~O 共 15 类。

最终决策：采用 A~O 共 15 类。

理由：用户已确认采用 A~O 共 15 类。

### 3. O 类蜂鸣器规则

前文写 A、N、O 类 1 秒脉冲，开发需求段落只写 A、N 类。

最终决策：O 类归入 A、N、O 严重故障同类，蜂鸣器长鸣。

理由：用户已确认 O 类属于严重故障，应归到 A、N、O 同类。

### 4. 自检策略

文档存在两套描述：

方案 A：智能状态识别与主动纠错。

- 根据 K1_EN/K2_EN/K3_EN 识别期望状态。
- 检查继电器状态并主动纠错。
- 检查接触器状态，不主动纠错。
- 检查温度安全。

方案 B：安全关断自检。

- 前置安全动作：确认所有通道关断。
- 输入检测：K1_EN、K2_EN、K3_EN 均为高电平。
- 反馈检测：9 路状态反馈均为低电平。
- 温度检测：NTC 均小于 60℃。

最终决策：采用“状态识别 + 安全复位 + 运行接管”的上电/复位策略。

这个策略不是简单的智能纠错，也不是一律安全关断。系统上电或复位后，先读取外部使能输入和继电器/接触器反馈，再根据组合状态决定是否接管、关断、开启或报警。

核心原则：

- 不因为 MCU 复位而擅自改变正在一致运行的现场状态。
- 不因为磁保持继电器掉电保持而把系统锁死。
- 只有在外部请求和实际反馈明确冲突时才报警禁止动作。
- 外部无使能请求时，允许把残留闭合的磁保持继电器安全关断。

正式流程：

```text
上电或复位
  ↓
读取 K1_EN / K2_EN / K3_EN
读取 Kx_1_STA / Kx_2_STA / SWx_STA
  ↓
判断外部使能请求数量
```

如果三路 `K_EN` 全为高电平：

- 若所有反馈均为低电平：进入待机状态。
- 若只有某一路反馈为打开：执行该通道安全关断脉冲，500ms 后复查；成功则进入待机，失败则触发 N 类异常。
- 若多路反馈为打开：触发 N 类异常，禁止自动处理。

如果只有一路 `K_EN` 为低电平：

- 若对应通道反馈为打开：接管当前运行状态，不输出继电器动作脉冲。
- 若所有反馈均为低电平：按正常流程打开对应通道。
- 若其他通道反馈为打开：触发异常，禁止自动切换。

如果多路 `K_EN` 为低电平：

- 触发 A 类严重故障。
- 禁止继电器动作。
- ALARM 输出低电平。
- 蜂鸣器长鸣。

## 新软件架构

### 1. 板级 IO 层

模块名：`board_io`

职责：

- 只读写 MCU 引脚。
- 不做消抖。
- 不判断业务含义。
- 不设置报警。

典型接口：

- `BoardIO_ReadK1EnRaw()`
- `BoardIO_ReadDcCtrlRaw()`
- `BoardIO_SetK1_1On()`
- `BoardIO_SetAlarmOutput()`

### 2. 输入采样层

模块名：`input_filter`

职责：

- 对 K1_EN、K2_EN、K3_EN、DC_CTRL 做采样。
- 先采样和判定 DC_CTRL，再决定是否允许处理 K_EN。
- 对 DC_CTRL 执行 50ms 间隔连续 3 次确认。
- 对 K_EN 执行 150ms 间隔连续 3 次确认。
- 输出稳定输入状态。
- 输出边沿事件。
- 管理 PLC 控制源有效性。
- 在 PLC 控制源异常或疑似异常时冻结 K_EN 控制事件。

边界规则：

- `DC_CTRL` 的采样、确认、疑似异常判断、恢复确认，都在 `input_filter` 层完成。
- `input_filter` 层负责输出 `PLC_CONTROL_VALID`、`PLC_CONTROL_SUSPECT_LOST`、`PLC_CONTROL_LOST`、`PLC_CONTROL_RESTORE_CONFIRM`。
- 当 PLC 控制源不是 `PLC_CONTROL_VALID` 时，`input_filter` 不发布新的 K_EN 开启或关闭事件。
- O 类异常的原始触发依据来自 `input_filter` 的 PLC 控制源状态。

触发方式建议：

- 不使用 EXTI 中断直接驱动业务。
- 采用固定周期轮询采样。
- 如果后续需要中断，只允许中断置位“需要采样”的标志，不允许在中断中执行业务。

理由：

- 原始计划使用中断，但实际硬件环境存在干扰。
- 轮询更容易做多次确认和抗干扰过滤。
- 所有输入统一进入同一个采样模型，便于排查。

输入处理顺序：

- `DC_CTRL` 的采样和判定必须早于 `K1_EN`、`K2_EN`、`K3_EN`。
- 每一轮输入处理都先更新 `DC_CTRL` 状态，再决定是否允许处理 K_EN。
- 如果 `DC_CTRL` 出现疑似异常，立即冻结 K_EN 控制事件。
- 如果 `DC_CTRL` 未确认正常，不允许发布 K_EN 开启或关闭事件。

K_EN 确认时间：

- 用户确认可以接受更长时间确认后再动作，但不得超过 500ms。
- K_EN 采用 150ms 间隔连续 3 次确认。
- K_EN 从疑似变化到确认动作的时间约为 450ms。
- K_EN 只在 PLC 控制源确认正常时才允许确认并发布事件。

DC_CTRL 与 K_EN 的关系：

- `DC_CTRL` 是 `K_EN` 的上级有效性条件。
- `DC_CTRL=0` 时，PLC 控制源正常，K_EN 事件可以生效。
- `DC_CTRL=1` 时，PLC 控制源异常，K_EN 事件全部无效。
- 一旦 DC_CTRL 原始采样出现疑似异常，应立即冻结 K_EN 事件，等待 DC_CTRL 确认。
- 如果 DC_CTRL 异常确认成立，触发 O 类严重故障，并继续忽略 K_EN。
- 如果 DC_CTRL 疑似异常未被确认，则解除 K_EN 冻结。

建议的 PLC 控制源状态：

```text
PLC_CONTROL_VALID              PLC 控制源正常，K_EN 可控制继电器
PLC_CONTROL_SUSPECT_LOST       DC_CTRL 疑似异常，立即冻结 K_EN
PLC_CONTROL_LOST               DC_CTRL 确认异常，触发 O 类严重故障
PLC_CONTROL_RESTORE_CONFIRM    DC_CTRL 恢复确认中，K_EN 暂不生效
```

DC_CTRL 确认要求：

- DC_CTRL 采用 50ms 间隔连续 3 次确认。
- DC_CTRL 从疑似异常到确认异常的时间约为 150ms。
- DC_CTRL 的疑似异常不需要等到 150ms 确认后才冻结 K_EN，而是第一次发现疑似异常就立即冻结 K_EN。
- 恢复时建议稳定确认 100ms~150ms，再重新同步 K_EN 和反馈状态。

### 3. 通道请求层

模块名：`channel_request`

职责：

- 根据稳定 K_EN 状态生成通道请求。
- 结合当前实际通道状态判断请求是否合法。
- 结合继电器动作忙闲状态决定是否允许生成新动作。
- 判断当前请求是打开、关闭还是无变化。
- 检测多路低电平冲突。
- 执行三通道互锁请求规则。
- 禁止通道间自动切换。

输入前提：

- 只接收 `input_filter` 输出的稳定状态。
- 需要接收当前实际通道状态。
- 需要接收继电器动作层忙闲状态。
- 只在 PLC 控制源确认正常时工作。
- 不采样、不确认、不二次判定 `DC_CTRL`。
- 如果 PLC 控制源不是 `PLC_CONTROL_VALID`，本层直接输出无新请求。
- 不直接读取 GPIO。
- 不直接驱动继电器。
- 不直接设置报警输出。

K_EN 组合规则：

```text
K1_EN K2_EN K3_EN    解析结果
  1     1     1      无通道请求，目标为全部关闭
  0     1     1      请求通道 1
  1     0     1      请求通道 2
  1     1     0      请求通道 3
  0     0     1      多路使能冲突，触发 A 类严重故障
  0     1     0      多路使能冲突，触发 A 类严重故障
  1     0     0      多路使能冲突，触发 A 类严重故障
  0     0     0      多路使能冲突，触发 A 类严重故障
```

当前实际通道状态：

```text
CHANNEL_ACTUAL_ALL_OFF     当前无通道打开
CHANNEL_ACTUAL_CH1_ON      当前通道 1 打开
CHANNEL_ACTUAL_CH2_ON      当前通道 2 打开
CHANNEL_ACTUAL_CH3_ON      当前通道 3 打开
CHANNEL_ACTUAL_UNKNOWN     当前反馈异常或状态不可信
```

动作忙闲规则：

- 当继电器动作层处于忙碌状态时，本层不生成新的打开动作请求。
- 忙碌状态包括 ON/OFF 脉冲输出期间、脉冲后的反馈等待期间、反馈判定尚未完成期间。
- 忙碌期间只更新稳定输入快照，不把输入变化排队为待执行动作。
- 继电器动作完成后，必须基于当时最新稳定输入和最新实际通道状态重新解析。
- 如果忙碌期间出现多路 K_EN 稳定有效，仍应触发 A 类严重故障，并禁止后续动作。

目标状态规则：

- 三路 `K_EN` 全高时，目标状态为全部关闭。
- 当前无通道打开且目标为全部关闭时，不输出继电器动作。
- 当前有通道打开且目标为全部关闭时，输出关闭当前通道请求。
- 当前无通道打开且目标为某一路通道时，输出打开目标通道请求。
- 当前已打开通道与目标通道一致时，不重复输出打开脉冲。
- 当前已打开通道与目标通道不一致时，判定为非法直接切换，触发 N 类严重故障。

通道切换规则：

- 系统不允许从通道 1 直接自动切换到通道 2 或通道 3。
- 系统不允许从通道 2 直接自动切换到通道 1 或通道 3。
- 系统不允许从通道 3 直接自动切换到通道 1 或通道 2。
- 用户或外部 PLC 必须先撤销当前通道使能，使三路 `K_EN` 全部为高电平。
- 当前通道完成关闭并确认反馈全部关闭后，才允许开启另一路通道。
- 如果当前通道仍处于打开反馈，而外部稳定请求变成另一路通道，系统判定为请求与反馈冲突，禁止继电器动作并交由安全报警层处理。
- 请求与反馈冲突按 N 类严重故障处理。
- 如果切换过程中短时间出现多路有效，但没有通过稳定确认，不触发 A 类异常。
- 如果多路有效通过稳定确认，触发 A 类严重故障。
- A 类严重故障解除后，不补执行故障期间曾经出现过的通道请求。
- A 类严重故障解除后，必须重新从当前稳定输入生成新的请求。
- 如果 N 类非法直接切换故障已经触发，解除条件为三路 `K_EN` 全部恢复高电平，并且实际反馈确认全部关闭。

输出：

目标状态输出：

- `CHANNEL_TARGET_ALL_OFF`
- `CHANNEL_TARGET_CH1`
- `CHANNEL_TARGET_CH2`
- `CHANNEL_TARGET_CH3`
- `CHANNEL_TARGET_CONFLICT`
- `CHANNEL_TARGET_UNKNOWN`

动作请求输出：

- `CHANNEL_ACTION_NONE`
- `CHANNEL_ACTION_OPEN_CH1`
- `CHANNEL_ACTION_CLOSE_CH1`
- `CHANNEL_ACTION_OPEN_CH2`
- `CHANNEL_ACTION_CLOSE_CH2`
- `CHANNEL_ACTION_OPEN_CH3`
- `CHANNEL_ACTION_CLOSE_CH3`
- `CHANNEL_ACTION_MULTI_ENABLE_CONFLICT`
- `CHANNEL_ACTION_ILLEGAL_SWITCH`

### 4. 继电器动作层

模块名：`relay_driver`

职责：

- 执行双路继电器 ON/OFF 脉冲。
- 同一通道的两路继电器必须同时动作。
- 保证输出低电平 500ms。
- 脉冲后等待 500ms。
- 向反馈判定层提供“可以检查反馈”的动作完成事件。
- 不直接判断报警。
- 不直接读取反馈。
- 不直接解析 K_EN。

建议实现：

- 使用非阻塞状态机。
- 状态机周期由主调度器推进。
- 禁止在主循环中使用长时间阻塞延时。
- 所有时间判断基于系统毫秒计数 `tick_ms`。
- 继电器动作时间使用 `APP_RELAY_PULSE_MS`。
- 反馈等待时间使用 `APP_RELAY_FEEDBACK_WAIT_MS`。

输入动作：

```text
RELAY_CMD_NONE
RELAY_CMD_OPEN_CH1
RELAY_CMD_CLOSE_CH1
RELAY_CMD_OPEN_CH2
RELAY_CMD_CLOSE_CH2
RELAY_CMD_OPEN_CH3
RELAY_CMD_CLOSE_CH3
```

输出控制规则：

```text
RELAY_CMD_OPEN_CH1   K1_1_ON  与 K1_2_ON  同时输出 500ms 低电平
RELAY_CMD_CLOSE_CH1  K1_1_OFF 与 K1_2_OFF 同时输出 500ms 低电平
RELAY_CMD_OPEN_CH2   K2_1_ON  与 K2_2_ON  同时输出 500ms 低电平
RELAY_CMD_CLOSE_CH2  K2_1_OFF 与 K2_2_OFF 同时输出 500ms 低电平
RELAY_CMD_OPEN_CH3   K3_1_ON  与 K3_2_ON  同时输出 500ms 低电平
RELAY_CMD_CLOSE_CH3  K3_1_OFF 与 K3_2_OFF 同时输出 500ms 低电平
```

安全输出规则：

- 初始化时，所有继电器驱动输出必须置为高电平。
- 空闲状态下，所有继电器驱动输出必须保持高电平。
- 当前动作只允许对应通道对应方向的两路输出拉低。
- 当前动作无关的其他 10 路继电器驱动输出必须保持高电平。
- 脉冲结束时，必须先恢复所有继电器驱动输出为高电平，再进入反馈等待。
- 如果状态机复位、命令非法或发生内部状态异常，必须立即恢复所有继电器驱动输出为高电平。

拒绝条件：

- `relay_driver` 非空闲时拒绝新命令。
- 忙碌拒绝只返回 `RELAY_RESULT_BUSY`，不直接触发报警。
- 输入命令为 `RELAY_CMD_NONE` 时不动作。
- 输入命令不在合法枚举范围内时拒绝，并恢复全部输出高电平。
- 拒绝命令不触发报警，只返回拒绝状态，由上层决定如何处理。

忙碌定义：

- 低电平脉冲输出期间为忙碌。
- 脉冲结束后的 500ms 反馈等待期间为忙碌。
- 动作完成事件尚未被上层取走前，状态机仍可视为忙碌，防止重复发起动作。

完成事件：

- 脉冲结束并等待 500ms 后，输出 `RELAY_RESULT_READY_FOR_FEEDBACK`。
- 完成事件必须包含通道号和动作方向。
- 反馈判定层根据该事件检查对应继电器反馈和接触器反馈。
- 本层不判断反馈是否正确，因此不直接产生 B~J 或 N 类异常。

状态示例：

- `RELAY_IDLE`
- `RELAY_PULSE_START`
- `RELAY_PULSE_ACTIVE`
- `RELAY_WAIT_FEEDBACK`
- `RELAY_DONE`
- `RELAY_REJECTED`

推荐接口：

```c
void RelayDriver_Init(void);
void RelayDriver_Task(uint32_t tick_ms);
RelayDriverResult_t RelayDriver_Start(RelayCommand_t command, uint32_t tick_ms);
uint8_t RelayDriver_IsBusy(void);
uint8_t RelayDriver_TakeDoneEvent(RelayDoneEvent_t *event);
void RelayDriver_ForceAllOutputsInactive(void);
```

### 5. 反馈判定层

模块名：`feedback_monitor`

职责：

- 读取继电器反馈和接触器反馈。
- 生成三通道实际状态。
- 根据期望状态判断哪个反馈异常。
- 判断通道打开或关闭动作是否反馈成功。
- 输出反馈异常位，不直接驱动报警输出。

反馈电平规则：

- 继电器反馈高电平有效。
- 接触器反馈高电平有效。
- 通道打开成功时，目标通道两路继电器反馈和一路接触器反馈必须全部为高电平。
- 通道打开成功时，非目标通道的六个反馈点必须全部为低电平。
- 通道关闭成功时，九个反馈点必须全部为低电平。
- 任意一个反馈点不符合预期，都必须输出对应 B~J 异常依据。

反馈异常映射：

```text
K1_1_STA 异常 -> B 类异常
K2_1_STA 异常 -> C 类异常
K3_1_STA 异常 -> D 类异常
K1_2_STA 异常 -> E 类异常
K2_2_STA 异常 -> F 类异常
K3_2_STA 异常 -> G 类异常
SW1_STA   异常 -> H 类异常
SW2_STA   异常 -> I 类异常
SW3_STA   异常 -> J 类异常
```

动作后反馈检查：

```text
打开通道 1：
  K1_1_STA=1，K1_2_STA=1，SW1_STA=1
  K2_1_STA=0，K2_2_STA=0，SW2_STA=0
  K3_1_STA=0，K3_2_STA=0，SW3_STA=0

关闭通道 1：
  九路反馈全部为 0

打开通道 2：
  K2_1_STA=1，K2_2_STA=1，SW2_STA=1
  K1_1_STA=0，K1_2_STA=0，SW1_STA=0
  K3_1_STA=0，K3_2_STA=0，SW3_STA=0

关闭通道 2：
  九路反馈全部为 0

打开通道 3：
  K3_1_STA=1，K3_2_STA=1，SW3_STA=1
  K1_1_STA=0，K1_2_STA=0，SW1_STA=0
  K2_1_STA=0，K2_2_STA=0，SW2_STA=0

关闭通道 3：
  九路反馈全部为 0
```

反馈异常后的保护规则：

- 一旦出现 B~J 任意反馈异常，系统不再接受新的通道动作指令。
- 反馈异常期间，`channel_request` 不应继续生成新的开关通道动作。
- B~J 异常允许自动解除，但必须满足正常一致状态。
- 只有三个通道的继电器反馈和接触器反馈恢复到正常一致状态后，才允许解除反馈异常保护。
- 反馈正常一致状态由当前系统目标决定：目标为某一路打开时，该通道三路反馈为高，其他通道反馈为低；目标为全部关闭时，九路反馈全部为低。
- B~J 异常解除后，不补执行异常期间曾经出现过的旧请求，必须重新稳定输入后再响应。
- B~J 异常解除后，系统根据当前稳定 K_EN 和当前实际反馈重新接管状态。
- 如果当前稳定 K_EN 与当前实际反馈一致，系统只接管当前状态，不重复输出继电器动作脉冲。
- 如果当前稳定 K_EN 请求另一路，而当前实际反馈显示已有通道打开，系统按非法直接切换处理，触发 N 类严重故障。

不负责：

- 不直接设置报警。
- 不直接驱动继电器。
- 不直接决定是否接受新的 K_EN 指令，只输出反馈保护状态供上层使用。

### 6. 安全报警层

模块名：`safety_manager`

职责：

- 统一管理 A~O 异常。
- 统一设置异常。
- 统一解除异常。
- 统一提供报警优先级。
- 提供“当前是否存在任意异常”状态给 ALARM 输出使用。
- 提供“当前最高异常等级”状态给蜂鸣器输出使用。
- 提供“当前是否允许新的继电器动作”状态给上层控制逻辑使用。

异常数量：

- 按 A~O 共 15 类实现。

异常保持规则：

- A~O 使用异常位图管理，多个异常可以同时存在。
- 新异常置位时，不覆盖已经存在的其他异常。
- 某个异常满足解除条件时，只清除该异常位，不影响其他仍存在的异常。
- 多个异常同时存在时，ALARM 仍只按“存在任意异常”输出。
- 多个异常同时存在时，蜂鸣器按最高异常等级输出。

动作禁止规则：

- A、N、O 严重故障存在时，禁止新的继电器动作请求。
- B~J 反馈异常存在时，禁止新的继电器动作请求。
- K~M 温度异常存在时，不禁止通道开关动作，只作为报警提示。
- O 类异常发生时，`input_filter` 已冻结 K_EN 事件，因此不会形成新的通道动作请求。
- O 类异常期间保持当前磁保持继电器状态，不主动关断。
- 任意影响动作安全的异常解除后，都不补执行异常期间曾经出现过的旧请求。
- 异常解除后，系统必须基于当前稳定输入和当前实际反馈重新判断。

报警解除：

- A：只有一路 K_EN 低电平或三路全高。
- A 解除后：不补执行故障期间的历史请求，必须重新稳定输入后再生成新请求。
- B~J：符合触发条件真值表。
- N：外部请求和实际反馈恢复到安全一致状态后允许解除。
- K~M：温度低于 58℃。
- O：DC_CTRL 恢复低电平。

### 7. 温度管理层

模块名：`temperature_manager`

职责：

- ADC DMA 数据处理。
- NTC 温度换算。
- 风扇 PWM 目标计算。
- 输出温度状态。
- 输出 K、L、M 温度异常依据。
- 输出风扇测速诊断状态。

温度输入：

- `NTC_1 -> PA0`，对应 K 类温度异常。
- `NTC_2 -> PA1`，对应 L 类温度异常。
- `NTC_3 -> PA2`，对应 M 类温度异常。

风扇控制：

- 风扇 PWM 只根据三路温度中的最高值控制。
- 三路最高温度大于等于 35℃ 时，风扇 PWM 目标为 95%。
- 三路最高温度下降到 33℃ 以下时，风扇 PWM 目标恢复为 50%。
- 风扇控制使用 2℃ 回差，避免温度在 35℃ 附近波动时 PWM 频繁切换。
- 任意一路温度大于等于 60℃ 时，风扇继续保持 95%。
- 风扇 PWM 输出参数使用 `APP_FAN_PWM_LOW_PERCENT` 和 `APP_FAN_PWM_HIGH_PERCENT`。

温度异常确认：

- K~M 属于低风险提示型异常。
- K~M 只输出报警信号和蜂鸣器提示，不限制通道开关动作。
- K~M 发生时，已经打开的通道不主动关断。
- K~M 发生时，新的通道开启或关闭仍按正常请求执行。
- 温度大于等于 60℃ 时，开始对应 K~M 异常确认。
- 温度下降到 58℃ 以下时，开始对应 K~M 异常解除确认。
- 温度异常确认周期为 500ms。
- 连续 3 次满足触发条件后，置位对应 K~M 异常依据。
- 连续 3 次满足解除条件后，解除对应 K~M 异常依据。
- 温度确认参数使用 `APP_TEMP_CONFIRM_PERIOD_MS` 和 `APP_TEMP_CONFIRM_COUNT`。
- 温度阈值使用 `APP_TEMP_FAN_HIGH_ON_C`、`APP_TEMP_FAN_LOW_ON_C`、`APP_TEMP_ALARM_SET_C`、`APP_TEMP_ALARM_CLEAR_C`。

NTC 异常诊断：

- 如果某一路 ADC 数值明显越界或 NTC 换算失败，该路视为 NTC 异常。
- NTC 异常不新增 A~O 异常编号。
- NTC 异常归入对应通道的 K、L、M 温度异常依据。
- OLED 显示层可以额外显示“NTC 异常”或“温度传感器异常”提示。
- NTC 异常同样不限制通道开关动作。

风扇测速：

- `FAN_SEN -> PC12`。
- 风扇测速异常不新增 A~O 异常编号。
- 第一版只作为诊断信息或 OLED 提示。
- 风扇测速状态不参与 ALARM 和 BEEP 输出。

报警置位：

- 温度模块不直接设置 K~M。
- K~M 由 `safety_manager` 根据温度状态统一设置。

### 8. 报警输出层

模块名：`alarm_output`

职责：

- 根据 `safety_manager` 的异常状态控制 ALARM。
- 根据最高优先级异常控制蜂鸣器。

ALARM 输出规则：

- ALARM 不区分异常等级。
- ALARM 为低电平有效，输出低电平表示有异常。
- ALARM 输出高电平表示无异常。
- 只要 A~O 任意异常存在，ALARM 持续输出低电平。
- 只有所有异常均解除后，ALARM 才恢复高电平。
- 所有异常解除后，ALARM 必须立即恢复高电平。
- ALARM 不做脉冲、不做闪烁、不按异常等级改变时序。

蜂鸣器输出规则：

- BEEP 为低电平有效，输出低电平表示蜂鸣器响。
- BEEP 输出高电平表示蜂鸣器不响。
- 蜂鸣器按当前最高异常等级决定输出时序。
- 多个异常同时存在时，蜂鸣器按最高等级输出。
- 最高异常等级变化时，蜂鸣器周期立即重置，并从“响”状态开始。
- 所有异常解除后，蜂鸣器必须立即停止，BEEP 恢复高电平。

故障等级：

1. 严重故障：A、N、O，蜂鸣器长鸣。
2. 紧急故障：B~J，蜂鸣器 100ms 响、100ms 停循环。
3. 一般故障：K~M，蜂鸣器 500ms 响、500ms 停循环。

蜂鸣器时序参数：

- 紧急故障响声时间：`APP_BEEP_URGENT_ON_MS = 100ms`。
- 紧急故障静音时间：`APP_BEEP_URGENT_OFF_MS = 100ms`。
- 一般故障响声时间：`APP_BEEP_GENERAL_ON_MS = 500ms`。
- 一般故障静音时间：`APP_BEEP_GENERAL_OFF_MS = 500ms`。

### 9. 显示层

模块名：`display_manager`

职责：

- 生成 OLED 显示内容。
- 控制 LOGO、进度条、待机界面、异常界面。
- 使用脏区刷新机制，只刷新变化区域。
- 只显示系统状态，不参与安全判定。
- 只读取各模块提供的状态快照，不直接读取 GPIO。
- 可借用旧工程 `Core/Src/minyerlogo.c` 中的 LOGO 字模和旧 OLED 驱动素材。
- 旧 LOGO 素材只作为图形来源，不直接使用旧版全屏显示尺寸。
- 不直接继承旧工程显示页面逻辑，界面重新排版。

界面规则：

- 屏幕为 128x64 单色 OLED，英文或缩写显示，不使用中文显示。
- 上电 LOGO：2 秒。
- 上电 LOGO 不采用全屏显示。
- 上电 LOGO 应在不影响辨识度的前提下适当缩小并居中显示。
- 上电 LOGO 建议宽度不超过 96 像素，高度不超过 32 像素。
- 如果旧 LOGO 字模尺寸过大，应重新缩放或重新生成适合 128x64 的单色位图。
- LOGO 显示期间不做阻塞等待。
- LOGO 显示期间允许后台完成基础初始化、时间基准准备、显示驱动准备和输入采样预热。
- LOGO 显示期间不输出继电器动作，不做最终自检结论，不触发业务报警动作。
- 自检界面：3 秒。
- 自检界面继续显示小 LOGO 或品牌标识。
- 自检界面 LOGO 应比上电 LOGO 更小，建议宽度不超过 48 像素，高度不超过 16 像素。
- 自检界面 LOGO 不占用进度条和自检项目文本区域。
- 自检界面下方显示进度条。
- 自检界面需要显示当前自检项目内容，使用英文或缩写，例如 `INPUT`、`FEEDBACK`、`TEMP`、`POWER`。
- 待机界面分三区：
  - 异常信息显示区。
  - 三通道状态区。
  - 温度和风扇转速区。
- OLED 运行界面需要同时显示三路温度，不能只显示最高温度。
- 三通道状态区分为三个等宽格，分别居中显示 `CH1`、`CH2`、`CH3`。
- 每个通道格宽度约 42 像素，高度根据页面布局确定。
- 通道格反显只表示实际反馈确认后的通道打开状态。
- 通道打开反馈确认后，该通道格使用反显形式显示。
- 通道关闭反馈确认后，该通道格使用正常字体显示。
- 继电器动作中和反馈等待中的短暂状态不单独显示。
- 外部 K_EN 请求状态不直接决定通道格是否反显。
- 通道异常时，可在对应通道格内增加 `!` 标记。
- 异常界面第一行显示最高优先级异常。
- 异常界面保留通道状态、三路温度和风扇状态摘要。
- 多个异常同时存在时，按最高等级优先显示；同等级多个异常每 1 秒轮询显示一个。
- OLED 显示不决定报警优先级，报警优先级由 `safety_manager` 提供。

刷新规则：

- 禁止主循环中无差别全屏刷新。
- 默认采用脏区刷新，只刷新发生变化的显示区域。
- 页面切换、LOGO 显示、首次进入某界面时允许整屏绘制一次。
- 通道状态变化时，只刷新三通道状态区。
- 温度或风扇数据变化时，只刷新温度和风扇信息区。
- 异常变化或异常轮询时，只刷新异常信息区。
- 自检进度变化时，只刷新进度条和当前自检项目区域。
- 如果脏区数量过多或页面布局完全变化，可以合并为较大的局部区域刷新，但应避免连续全屏刷新造成闪烁。

建议页面草图：

```text
LOGO:
        [MINYER]

SELF TEST:
  [MYR]       SELF TEST
  INPUT       [####------]

STANDBY:
  OK          STBY
  [ CH1 ][ CH2 ][ CH3 ]
  T1 32  T2 31  T3 33
  FAN 50%  RPM 1200

RUNNING:
  OK          RUN CH1
  [>CH1<][ CH2 ][ CH3 ]
  T1 36  T2 32  T3 33
  FAN 95%  RPM 1800

ALARM:
  ALM B K1_1
  [>CH1!][ CH2 ][ CH3 ]
  T1 36  T2 32  T3 33
  FAN 95%  RPM 1800
```

### 10. 看门狗层

模块名：`watchdog_manager`

职责：

- IWDG 初始化。
- 喂狗策略。
- 复位原因记录。
- 主循环运行健康检查。
- 内部不可恢复错误处理。

建议策略：

- 调试阶段默认关闭看门狗，避免影响断点调试。
- 生产阶段通过 `APP_WATCHDOG_ENABLE = 1` 启用看门狗。
- 看门狗是否真正启动只由编译配置控制，不在运行中由业务异常临时开关。
- 看门狗喂狗必须集中在 `watchdog_manager`，禁止各处散落喂狗。
- A~O 业务异常不直接停止喂狗。
- A、N、O 严重故障只通过报警和动作禁止处理，不作为看门狗复位理由。
- B~J 反馈异常只通过报警和动作禁止处理，不作为看门狗复位理由。
- K~M 温度异常只作为报警提示，不作为看门狗复位理由。
- O 类 PLC 控制源异常期间保持当前继电器状态，不因为 O 类异常停止喂狗。
- 看门狗只用于处理软件卡死、主循环失控、调度异常或内部不可恢复错误。
- LOGO、自检、继电器动作等待期间，只要主循环仍按非阻塞方式正常运行，就必须继续按规则喂狗。

喂狗条件：

- 主循环正常运行。
- 系统 tick 正常更新。
- 关键任务均在允许周期内被调用。
- 未进入内部不可恢复错误状态。
- 未调用 `WatchdogManager_Panic()`。

不喂狗条件：

- 主状态机进入不可恢复内部错误。
- 主循环长时间未完成。
- 关键模块长时间未被调度。
- 系统 tick 停止或明显异常。
- 内部状态结构损坏或出现无法恢复的软件错误。
- 代码主动调用 `WatchdogManager_Panic()`。

时间约束：

- 看门狗超时时间建议为 `APP_WATCHDOG_TIMEOUT_MS = 3000ms`。
- 集中喂狗周期建议为 `APP_WATCHDOG_FEED_PERIOD_MS = 500ms`。
- 主循环单次运行耗时超过 `APP_MAIN_LOOP_WARN_MS = 50ms` 时记录诊断信息。
- 主循环单次运行耗时超过 `APP_MAIN_LOOP_MAX_MS = 200ms` 时视为调度异常依据。
- 所有模块不得使用长时间阻塞延时破坏主循环调度。
- 继电器 500ms 脉冲、反馈 500ms 等待、LOGO 2 秒、自检 3 秒均必须使用非阻塞状态机实现。

接口原则：

- 其他模块不得直接喂狗。
- 其他模块只能上报健康状态或请求进入内部错误状态。
- `watchdog_manager` 在主循环末尾统一判断是否喂狗。
- 如果看门狗导致复位，重启后应记录或显示上次复位原因为看门狗复位。
- 看门狗复位原因只作为诊断信息或 OLED 提示，例如 `WDT RST`。
- 看门狗复位原因不作为 A~O 业务异常，不拉低 `ALARM`，不触发蜂鸣器，也不改变继电器接管策略。

### 11. 维护功能层

模块名：`maintenance_manager`

职责：

- 管理 KEY1、KEY2 两个本地维护按键输入。
- 管理调试串口日志调出、分页、翻页和清空。
- 管理 Flash 日志记录和运行记录等非易失数据。
- 预留 PC15 运行心跳输出功能。
- 预留 RS485 维护通信接口。

边界规则：

- 维护功能不得绕过安全报警层直接强行动作继电器。
- 维护功能不得绕过输入采样、通道请求、反馈判定等核心安全链路。
- 两颗维护按键不用于蜂鸣器静音。
- 两颗维护按键不用于强制清除当前真实存在的 A~O 异常。
- 清空日志只清除历史记录，不清除当前故障状态，不改变 ALARM、蜂鸣器和继电器状态。
- PC15 当前保持空闲，不作为 KEY3 使用。
- PC15 后续计划作为 1kHz 运行心跳输出，用于通知客户 PLC 系统主循环仍在运行；该功能放到最后阶段单独讨论和实现。
- RS485 后续如果启用，也只能提交维护请求，不直接操作底层 IO。

当前阶段策略：

- KEY1、KEY2 采用轮询采样，不使用 EXTI 中断。
- KEY1、KEY2 按键确认时间建议为 `APP_MAINT_KEY_CONFIRM_MS = 50ms`。
- KEY1、KEY2 通过短按和长按区分日志操作。
- KEY1、KEY2 长按触发时间为 `APP_MAINT_KEY_LONG_MS = 3000ms`。
- 长按必须在消抖确认后持续满 3 秒才触发。
- 如果按键按下不足 3 秒就释放，不触发长按动作，也不转为短按动作。
- KEY1、KEY2 只服务日志串口操作，不作为核心控制依赖。
- 日志功能第一版启用，主要通过调试串口调出。
- 日志产品上限按 `APP_LOG_MAX_RECORDS = 2000` 条设计。
- 使用循环日志策略，超过 2000 条后新日志覆盖最旧日志。
- 日志每页输出 `APP_LOG_PAGE_SIZE = 50` 条，输出一页后暂停，等待按键翻页。
- 日志时间戳采用 `uint32_t`，单位为秒，从每次上电启动开始计时。
- 日志按事件记录，不做每秒定时写入；“秒”只作为事件时间戳单位。
- 日志存储使用 W25Q128JVSIQ Flash。该器件容量为 128Mbit，即 16MB，实际可容纳的日志数量远大于 2000 条；2000 条是为了产品维护、翻页查看和磨损控制设定的保守上限。
- 日志 Flash 起始地址、占用扇区数量、磨损均衡和掉电保护策略在 `log_manager` 编码阶段单独细化。
- RS485 先保留模块边界和接口，不进入第一版核心动作链路。

建议按键功能：

- KEY1 短按：调出日志，或回到上一页日志。
- KEY2 短按：翻到下一页日志。
- KEY1 长按：退出日志分页输出状态。
- KEY2 长按：清空日志记录。
- 日志分页输出期间，系统主循环必须继续运行，不得因为等待翻页阻塞安全逻辑、报警逻辑或看门狗喂狗。

日志记录内容：

- 每次上电启动。
- 上一次复位原因。
- 看门狗复位后的软件原因诊断结果。
- 每一次通道动作状态，例如通道 1 打开、通道 1 关闭、通道 2 打开、通道 2 关闭、通道 3 打开、通道 3 关闭。
- 只记录实际发生动作的目标通道，不记录其他未动作通道的维持状态。
- 每一次继电器动作失败。
- 每一次 A~O 异常产生。
- 每一次 A~O 异常解除。
- 每一次日志清空操作。
- 执行日志清空后，应立即写入一条新的 `LOG_CLEARED` 记录，表示日志是被维护操作清空，而不是异常丢失。
- 温度正常变化、风扇 PWM 变化、风扇转速变化不写入日志。
- 只有当温度或风扇相关问题触发 A~O 中的异常事件时，才按异常产生或异常解除记录。

建议日志记录格式：

```c
typedef struct
{
    uint32_t time_s;
    uint16_t event_id;
    uint16_t param;
    uint32_t detail;
} AppLogRecord_t;
```

- `time_s`：本次上电后的运行秒数。
- `event_id`：事件类型，例如上电、通道动作、异常产生、异常解除、看门狗复位、日志清空。
- `param`：事件对象，例如通道号、故障编号或复位原因编号。
- `detail`：补充信息，例如动作方向、反馈快照、模块心跳快照或错误码。
- 日志输出到调试串口时再转换为可读文本，Flash 内部优先保存固定长度二进制记录。

看门狗复位原因记录：

- MCU 可以在启动时读取硬件复位标志，判断上一次是否为 IWDG 复位。
- 如果上一次是 IWDG 复位，`watchdog_manager` 应提供最后一次内部停止喂狗原因。
- 停止喂狗原因建议在停止喂狗前写入一份小型复位诊断记录，内容包括软件原因编号、发生时刻、主状态机状态和关键模块心跳状态。
- 下次启动后，日志系统应记录 `WDT_RESET` 事件，并附带软件原因编号。
- 如果无法读取到明确的软件原因，则记录为 `WDT_REASON_UNKNOWN`。
- 看门狗复位原因仍只作为日志和诊断信息，不作为 A~O 业务异常。

PC15 运行心跳预留：

- PC15 暂不作为按键输入，不参与当前维护功能。
- 后续计划在程序正常运行、主循环持续调度时输出 1kHz 脉冲。
- 如果程序死机、复位或主循环停止，PC15 脉冲应停止，使外部 PLC 能识别控制器异常。
- 该功能需要最后单独讨论：脉冲由定时器硬件输出还是由主循环健康门控输出、异常期间是否关闭、复位期间硬件默认态如何保证。

### 12. 应用状态机

模块名：`app_controller`

系统状态：

- `APP_STATE_BOOT`
- `APP_STATE_LOGO`
- `APP_STATE_SELF_TEST`
- `APP_STATE_STANDBY`
- `APP_STATE_RUNNING`
- `APP_STATE_ALARM`
- `APP_STATE_ERROR`

状态职责：

- `APP_STATE_BOOT`：只做硬件默认安全态、模块初始化、复位原因读取和日志系统启动准备。
- `APP_STATE_LOGO`：显示 2 秒 LOGO，同时允许基础初始化、时间基准准备、输入采样预热、温度采样预热和复位原因日志记录。
- `APP_STATE_SELF_TEST`：显示 3 秒自检界面和进度条，完成输入、反馈、温度、外部控制电源和继电器残留状态识别。
- `APP_STATE_STANDBY`：无通道打开、无动作执行，等待稳定有效的单路通道请求。
- `APP_STATE_RUNNING`：已有通道打开并由反馈确认，持续监控输入、电源、反馈、温度和异常。
- `APP_STATE_ALARM`：存在 A~O 异常。是否允许继续动作由 `safety_manager` 决定，状态名只表示当前存在异常。
- `APP_STATE_ERROR`：软件内部不可恢复错误或状态机非法状态，进入后禁止继电器新动作，并请求 `watchdog_manager` 停止喂狗或进入 panic。

LOGO 阶段规则：

- 允许初始化板级 IO、OLED、Flash、日志、输入采样、温度采样、看门狗管理器和维护按键模块。
- 允许读取并记录上一次复位原因。
- 允许输入采样和温度采样预热，但不发布继电器动作请求。
- 禁止输出继电器 ON/OFF 动作。
- 禁止形成最终自检结论。
- 禁止触发业务报警输出结论。
- 如果初始化发现内部不可恢复错误，可直接进入 `APP_STATE_ERROR`。

自检阶段规则：

- 自检阶段必须继续主循环调度和看门狗喂狗。
- 自检界面显示小 LOGO、进度条和当前自检项目。
- 自检项目包括输入稳定状态、九路反馈状态、温度采样有效性、PLC 控制源状态、磁保持继电器残留状态识别。
- 如果外部无使能请求且存在单一路通道残留打开，允许执行该通道安全关断流程。
- 如果外部请求与实际反馈一致，接管当前运行状态，不重复输出继电器动作。
- 如果发现多路反馈打开、外部请求与反馈冲突、反馈无法安全恢复，触发 N 类异常并进入 `APP_STATE_ALARM`。
- 自检通过后，无通道打开进入 `APP_STATE_STANDBY`，有通道接管进入 `APP_STATE_RUNNING`。

运行阶段规则：

- `APP_STATE_STANDBY` 和 `APP_STATE_RUNNING` 都由当前稳定输入、反馈状态和安全层许可决定是否发起继电器动作。
- A、B~J、N、O 等禁止动作类异常存在时，不发起新的继电器动作。
- K~M 温度类一般故障存在时，仍允许按稳定请求执行通道动作。
- O 类 PLC 控制源异常存在时，输入采样层冻结 K_EN 事件，应用状态机保持当前继电器状态。
- 所有 A~O 异常解除后，应用状态机基于当前稳定输入和当前反馈重新接管，不补执行旧请求。

主循环：

```c
while (1) {
    App_RunOnce();
}
```

`App_RunOnce()` 顺序：

1. 更新时间。
2. 读取原始 IO 快照。
3. 输入采样和维护按键采样。
4. 温度更新和风扇测速更新。
5. 继电器状态机推进。
6. 反馈快照更新。
7. 安全报警证据更新。
8. 应用状态机更新。
9. 通道请求解析和动作许可判断。
10. 报警输出和蜂鸣器更新。
11. OLED 显示更新。
12. 日志和调试串口输出推进。
13. 看门狗处理。

调度原则：

- `App_RunOnce()` 不得包含阻塞延时。
- LOGO 2 秒、自检 3 秒、继电器 500ms 脉冲、反馈 500ms 等待、日志分页等待都必须由状态机计时完成。
- 所有模块只读写自己的内部状态，通过快照、事件或明确接口交换信息。
- 继电器动作必须经过 `safety_manager` 许可和 `relay_driver` 忙碌检查。
- `watchdog_manager` 必须在主循环末尾统一判断是否喂狗。

## 讨论路线与架构对应关系

最初列出的模块顺序是讨论顺序，本文“新软件架构”章节是代码分层顺序。两者不要求完全同序，但必须能一一对应。

| 讨论模块 | 架构模块 | 说明 |
|---|---|---|
| 系统状态机 / 上电复位流程 | `app_controller` | 作为总控状态机，放在架构末尾，是因为它调度前面所有模块。 |
| 输入采样模块 | `input_filter` | 已细化 K_EN、DC_CTRL、采样周期和冻结规则。 |
| 通道请求判断模块 | `channel_request` | 已细化三路互锁、禁止自动切换、目标状态和动作请求。 |
| 继电器动作模块 | `relay_driver` | 已细化 500ms 脉冲、500ms 反馈等待、忙碌拒绝和安全输出。 |
| 反馈判定模块 | `feedback_monitor` | 已细化九点反馈判定、B~J 映射和反馈异常保护。 |
| 安全报警模块 | `safety_manager` | 后续继续讨论 A~O 触发、解除、保持和优先级。 |
| 报警输出和蜂鸣器模块 | `alarm_output` | ALARM 按任意异常持续输出，BEEP 按最高故障等级输出。 |
| 温度和风扇模块 | `temperature_manager` | 温度采集、阈值、风扇 PWM 和风扇测速归在同一层讨论。 |
| OLED 显示模块 | `display_manager` | 负责 LOGO、自检、待机、运行、报警界面。 |
| 看门狗模块 | `watchdog_manager` | 负责调试和生产阶段喂狗策略。 |
| 维护功能模块 | `maintenance_manager` | 负责 KEY、Flash、日志、RS485 预留等维护能力。 |
| 板级 IO 层 | `board_io` | 这是代码实现必须有的底层，不属于最初业务讨论模块，但所有模块都通过它访问硬件。 |

## 关键设计结论

1. 新项目应使用轮询采样作为主输入机制，不再让中断直接触发业务动作。
2. 继电器动作必须按原始需求恢复为 500ms 脉冲和 500ms 反馈检查。
3. 看门狗必须重新规划，不继承旧代码中分散喂狗和临时禁用的做法。
4. 上电/复位采用“状态识别 + 安全复位 + 运行接管”方案。
5. 旧代码只能借用独立资源，不借用架构。
