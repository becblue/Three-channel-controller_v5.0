## 复盘目的

原始 `DOC/README.md` 是最早的代码实现依据，优先级高于我根据当前代码反推出来的理解。

当前代码已经经过多轮修改，存在新旧逻辑混杂的情况。因此重构时不能只照着当前代码搬迁，必须同时对照原始需求，区分：

- 原始需求是什么；

- 当前代码和原始需求哪里不一致；
- 哪些不一致是明确修复，哪些是后续改坏的。

## 原始需求中的核心系统

原始项目核心是三通道高压切换箱控制系统，包含：

- 三路切换使能输入：`K1_EN`、`K2_EN`、`K3_EN`。
- 双路磁保持继电器驱动：每通道两路继电器。
- 双路继电器状态反馈。
- 接触器状态反馈。
- OLED 状态显示。
- 外部 DC24V 电源异常监控。
- 一路 ALARM 报警输出。
- 蜂鸣器报警输出。
- 三路 NTC 温度监控。
- 温度联动风扇控制。
- 预留 Flash、按键、RS485。

这说明外部 DC24V 检测只是安全输入之一，不是项目核心。

## 必须保留的硬件电平规则

### K_EN 输入

- `K1_EN -> PB9`
- `K2_EN -> PB8`
- `K3_EN -> PA15`
- 下降沿表示通道开启请求。
- 上升沿表示通道关闭请求。
- 原始需求中要求开启中断，优先级 4。

当前代码改成轮询模式，这可能是后续为了抗干扰或稳定性做的修改。重构时可以继续采用轮询，但必须保留原始行为语义：

- 低电平表示对应通道使能请求有效。
- 高电平表示对应通道使能请求无效或关闭请求。
- 原始 README 写明所有信号检测采用 50ms 间隔连续 3 次确认机制。

2026-06-17 最终修订口径：

- `DC_CTRL` 仍采用 50ms 间隔连续 3 次确认。
- `K_EN` 改为 150ms 间隔连续 3 次确认，确认时间约 450ms，且不得超过 500ms。
- `DC_CTRL` 是 `K_EN` 的前置有效性条件，必须先判定 `DC_CTRL`，再处理 `K_EN`。

### 继电器输出

所有继电器控制输出均为低电平有效，初始高电平。

原始需求明确要求：

- 开启动作输出 500ms 低电平脉冲。
- 关闭动作输出 500ms 低电平脉冲。
- 脉冲结束后延时 500ms 再检查状态反馈。

当前代码中异步模式使用了较短的 100ms 脉冲和 100ms 反馈延时，这和原始 README 的 500ms 要求不一致。重构时不能默认继承当前 100ms 参数，必须作为待确认差异处理。

### 继电器和接触器反馈

继电器反馈和接触器反馈均为高电平有效，初始低电平。

通道打开时，目标通道的两路继电器反馈和对应接触器反馈应为高电平。

通道关闭时，对应反馈应为低电平。

### 外部电源监控

原始 README 明确写明：

- `DC_CTRL -> PB5`
- 上升沿使能。
- 内部上拉。
- 优先级 2。

附录中进一步明确：

- `DC_CTRL = 0`：有 DC24V 供电，正常状态。
- `DC_CTRL = 1`：没有 DC24V 供电，异常状态。
- 电源中断条件是 `DC_CTRL` 从低电平变为高电平。
- O 类异常解除条件是 `DC_CTRL` 恢复低电平。

这证明当前 `gpio_control.c` 中 `stable_state == 0` 触发电源异常快速回调，与原始需求方向不一致。

## 原始报警规则

异常定义为 A-O，共 15 类。

虽然 README 中有一处写“14种异常类型”，但后续明确列出了 A-O，因此应以 A-O 共 15 类为准。

报警输出规则：

- 任意异常激活时，`ALARM -> PB4` 持续输出低电平。
- K-M 温度异常：蜂鸣器持续低电平。
- B-J 状态反馈异常：蜂鸣器 50ms 间隔脉冲。
- A、N、O 类异常：蜂鸣器 1 秒间隔脉冲。

注意：README 前文有一处只写了 A、N 类 1 秒脉冲，后文和 O 类附录明确把 O 类归入 A、N、O 类。

2026-06-17 用户最终确认：

- A、N、O 类属于严重故障，蜂鸣器长鸣。
- B~J 类属于紧急故障，蜂鸣器 50ms 循环脉冲。
- K~M 类属于一般故障，蜂鸣器 1 秒脉冲。

## 原始报警解除规则

### A 类异常

当 K1_EN、K2_EN、K3_EN 中只能有一路低电平，或三路均为高电平时，A 类异常可解除。

### B-J、N 类异常

需要持续检测：

- K1_EN
- K2_EN
- K3_EN
- K1_1_STA
- K1_2_STA
- K2_1_STA
- K2_2_STA
- K3_1_STA
- K3_2_STA
- SW1_STA
- SW2_STA
- SW3_STA

当状态符合触发条件真值表时，报警解除。

### K-M 类异常

温度下降到 58℃ 以下后解除报警。

### O 类异常

原始附录明确：DC_CTRL 恢复低电平后自动解除。

我之前的当前逻辑文档中没有把 O 类解除条件单独写清楚，需要补充。

## 自检需求存在历史差异

原始 README 中自检描述有两套痕迹：

### 较早的智能状态识别版本

描述为：

1. 根据 K1_EN/K2_EN/K3_EN 识别期望状态。
2. 检查继电器反馈并主动纠错。
3. 检查接触器反馈，不主动纠错。
4. 检查温度安全。

### 后续开发需求版本

描述为：

1. 前置安全动作：通道关断确认。
2. 输入信号检测：K1_EN、K2_EN、K3_EN 均为高电平。
3. 状态反馈检测：9 路反馈均为低电平。
4. 温度安全检测：3 路 NTC 均小于 60℃。
5. 任一失败产生 N 类异常。

当前代码更接近“智能状态识别 + 继电器纠错 + 接触器检查 + 温度检测”的版本。

重构时不能贸然选其中一个作为最终规则，必须把自检需求列为“待确认设计点”。

## 主循环调度差异

原始 README 中描述的调度周期包括：

- 继电器：10ms。
- 温度监控：100ms。
- 显示更新：200ms。
- 安全监控：500ms。

当前代码实际存在：

- 继电器异步状态机 5ms 和 1ms 重复调度。
- 安全监控 100ms 调度。

这属于当前实现和原始需求不一致的地方。重构时应重新确定调度周期，而不是直接继承当前代码。

## 温度控制要求

原始 README 明确：

- 温度 < 35℃：风扇 PWM 50%。
- 温度 >= 35℃：风扇 PWM 95%。
- 温度 >= 60℃：触发 K、L、M 对应异常。
- 温度下降时需要 2℃ 回差。

重构时 `temperature_manager` 应输出温度状态和风扇控制结果，K-M 报警应由 `safety_manager` 统一置位。
